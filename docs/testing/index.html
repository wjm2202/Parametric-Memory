Training Your Memory

AI assistants forget everything between sessions. They repeat mistakes you already corrected. They lose track of your architecture decisions. They guess instead of remembering what worked.

Parametric-Memory fixes this. This guide shows you how to train a memory that makes your AI assistant genuinely better over time — not through fine-tuning, but through stored facts, corrections, and proven processes that load at the start of every session.

1

Correcting AI Behaviour

The highest-value thing you can store is a correction. When your AI makes a mistake and you tell it the right way, that correction should persist forever.

The problem: You tell Claude "don't guess at the root cause, diagnose it with evidence." It works for that session. Next session, it guesses again.

The fix: Store the correction as aprocedure atom. Procedures are loaded at the start of every session and applied before any action.

Storing steering corrections

Each correction becomes a durable rule the AI checks before acting.

v1.procedure.never_guess_root_cause_always_diagnose_with_evidence// from user correction
v1.procedure.always_run_full_test_suite_before_claiming_done
v1.procedure.read_full_error_message_before_proposing_fix
v1.procedure.do_not_edit_config_files_without_explicit_approval
v1.procedure.fix_errors_directly_minimize_apology
v1.procedure.when_blocked_ask_user_not_overengineer_workaround

Reinforcing the correction arc

Train the Markov chain: wrong behaviour → correction → right behaviour.

train([
"v1.event.user_corrected_guessing_behaviour",
"v1.procedure.never_guess_root_cause_always_diagnose_with_evidence",
"v1.event.fix_found_by_systematic_diagnosis"
])

Examples of corrections worth storing

"Don't guess, diagnose the root cause"→ v1.procedure.diagnose_root_cause_not_guess
"Run all tests, not just the ones you think matter"→ v1.procedure.always_run_full_test_suite
"Don't say failures are pre-existing"→ v1.procedure.all_tests_must_pass_for_pr
"Stop apologising and fix it"→ v1.procedure.fix_errors_directly_minimize_apology
"Read the actual error message"→ v1.procedure.read_full_error_before_proposing_fix
"If you can't do something, ask me — don't build a complex workaround"→ v1.procedure.when_blocked_ask_user_not_overengineer
"Always list files and reasons before requesting delete"→ v1.procedure.list_files_with_reasons_before_delete

Storing Project Details & Architecture

Every time your AI asks "what framework are you using?" or "where is the config?" — that's a fact that should have been in memory.

Stack and configuration facts

Store once, available in every session. No more re-explaining your setup.

v1.fact.stack_is_TypeScript_Fastify_LevelDB
v1.fact.test_framework_is_Vitest_with_671_tests
v1.fact.deployment_target_is_GitHub_Pages_static_export
v1.fact.node_version_20_required
v1.fact.mmpm_server_runs_on_port_3000
v1.fact.website_uses_Next14_Tailwind_Framer_Motion

Architecture decisions and constraints

Why you chose X over Y. Constraints that shaped the design.

v1.fact.chose_LevelDB_over_SQLite_for_append_only_writes
v1.fact.merkle_tree_is_incremental_not_rebuilt_each_time
v1.fact.csr_matrix_chosen_for_sparse_transition_storage
v1.fact.mcp_uses_stdio_bridge_not_http_direct
v1.relation.shard_worker_uses_injectable_clock_for_testing
v1.fact.output_export_means_no_server_side_api_routes

Naming conventions and patterns

How your codebase is structured. Patterns to follow.

v1.fact.atom_naming_schema_v1_type_value_snake_case
v1.fact.test_files_colocated_in_src_tests_directory
v1.fact.all_routes_defined_in_server_ts_not_split
v1.fact.env_vars_documented_in_env_example

Pro tip: Use the relation type to link concepts. When the AI accesses one atom, the Markov engine predicts related ones — so storing v1.relation.ingestion_depends_on_write_ahead_log means that accessing anything about ingestion automatically surfaces the WAL context.

3

Sprint Progress Tracking

Use state atoms to track what's done, what's in progress, and what's next. Tombstone states when they're no longer true.

Active sprint state

Current work context that loads at session start.

v1.state.sprint_17_in_progress
v1.state.S17_0_scaffold_complete
v1.state.S17_1_landing_page_complete
v1.state.S17_2_why_page_complete
v1.state.S17_3_docs_in_progress// ← current task
v1.state.S17_4_contributing_page_pending

Completing a sprint item

When a task finishes, store the completion event and tombstone the old state.

checkpoint({
atoms: [
"v1.event.S17_3_docs_complete_dt_2026_03_08",
"v1.state.S17_4_contributing_page_next"
],
tombstone: [
"v1.state.S17_3_docs_in_progress"
],
train: [
"v1.state.S17_3_docs_in_progress",
"v1.event.S17_3_docs_complete_dt_2026_03_08",
"v1.state.S17_4_contributing_page_next"
]
})

Why tombstone? Without tombstoning, the AI will see both "docs in progress" and "docs complete" in memory and get confused. Tombstoning removes stale state so only the current truth exists. The tombstoned atom is preserved in the audit log for history — it's just removed from active recall.

Session start: where we left off

The bootstrap call returns current states, so the AI knows exactly where to resume.

// AI sees at session start:
state: sprint_17_in_progress
state: S17_0 thru S17_3 complete
state: S17_4_contributing_page_next
// → AI immediately knows: "We're building the Contributing page next."
4

Reusing Proven Solutions

When a debugging approach works, a deploy process succeeds, or a workaround solves a tricky problem — store the process as a procedure so it can be replayed exactly next time.

Storing proven debugging processes

Step-by-step processes that worked, named for searchability.

v1.procedure.debug_shard_worker_check_fake_timers_and_db_path_collisions
v1.procedure.fix_mcp_connection_verify_cwd_in_server_config_first
v1.procedure.restore_atoms_use_npm_run_restore_not_manual_curl
v1.procedure.fumadocs_v14_uses_layouts_slash_docs_not_layout
v1.procedure.fumadocs_css_import_is_style_dot_css_not_dist

Training the solution arc

Teach the Markov chain: when this problem occurs → use this solution → expect this outcome.

train([
"v1.event.shard_worker_tests_failing",
"v1.procedure.debug_shard_worker_check_fake_timers_and_db_path_collisions",
"v1.event.all_shard_worker_tests_passing"
])

What to store as procedures

Build and deploy sequences that work

Debugging approaches that found the root cause

Test strategies that caught real bugs

Configuration sequences for tools

Workarounds for known environment quirks

What NOT to reinforce

Lucky guesses that happened to work

Workarounds that masked the real problem

Approaches the user later corrected

Processes that only worked in a specific context

Unverified assumptions about root causes

5

Bootstrap: Context Injection

The session_bootstrap call is what makes everything work. It runs at the start of every session and injects your stored memory into the AI's context window.

What bootstrap returns

1
Procedures — corrections and proven processes. Applied before any action.
2
Facts — project architecture, stack, naming conventions.
3
States — current sprint progress, what's in-flight, what's next.
4
Relations — links between concepts for Markov-predicted context.
5
Predictions — Markov-weighted atoms likely relevant to the current objective.

Bootstrap with an objective

Pass the user's opening message as an objective to get targeted predictions.

session_bootstrap({
objective: "Fix the failing shard worker tests",
maxTokens: 1200
})
// Returns:
// - procedure: debug_shard_worker_check_fake_timers...
// - fact: shard_worker_uses_injectable_clock
// - state: sprint_14_shard_reliability
// - prediction: event.shard_worker_tests_failing → procedure.debug_shard...

What the AI does with bootstrap context

Priority order: procedures first (constraints), then facts, states, relations, predictions.

// 1. Check procedures — do any constrain my next action?
// "never_guess_root_cause" → I must diagnose with evidence
// "always_run_full_test_suite" → I must run all tests, not just targeted
// 2. Load facts — what do I know about this project?
// "stack_is_TypeScript_Fastify_LevelDB"
// "shard_worker_uses_injectable_clock_for_testing"
// 3. Check state — where are we in the sprint?
// "S17_3_docs_in_progress" → this is what we're working on
// 4. Use predictions as hints, not absolute truth

The key insight: Bootstrap doesn't just load raw data — the Markov engine predicts which atoms are most relevant to your current objective. If you've trained arcs like problem → solution → outcome, the engine will surface the right solution when it sees a similar problem. This is why training arcs matters — it teaches the memory which connections are meaningful.

The Memory Training Loop

Every session follows the same pattern. Over time, the memory compounds — corrections accumulate, solutions get reinforced, and the AI becomes measurably more effective at working on your project.

1

Load

Bootstrap loads procedures, facts, states, and predictions into the context window.

2

Work

As you work, save discoveries, corrections, and completions immediately — not just at session end.

3

Train

Train arcs after successful workflows. Tombstone stale states. Commit to disk.

What this solves

✗
"AI keeps making the same mistake"→ Store corrections as procedures
✗
"AI doesn't know my stack"→ Store architecture as facts
✗
"AI forgot where we left off"→ Track progress with states
✗
"AI reinvents solutions it already found"→ Store working processes as procedures
✗
"AI doesn't understand my codebase"→ Store relations between modules
✗
"AI context is always stale"→ Bootstrap injects fresh memory every session
