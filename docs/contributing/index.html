Contributing to Parametric-Memory

We welcome contributions from the community. Whether you're fixing a bug, adding a feature, or improving documentation — this guide covers everything you need to get started.

Before You Start

For anything beyond a small bug fix, open an issue first. Describe what you want to add or change and why. Architectural or scope questions are much cheaper to resolve before code is written.

Development Setup

Terminal
git clone https://github.com/wjm2202/Parametric-Memory.git
cd Parametric-Memory
npm run setup

npm run setup installs dependencies, compiles TypeScript, and creates .env from .env.example if it doesn't exist.

node dist/server.js    # start the server
npm test               # run all tests (671)
npm run typecheck      # type-check src + tests

Architecture Overview

How the modules connect. Gold is the API surface, teal is the core engine layer, silver is data/policy, and grey is infrastructure.

Project Layout

src/                    TypeScript source
  server.ts             Fastify HTTP server — all routes
  orchestrator.ts       Sharded LevelDB coordination
  ingestion.ts          Write-ahead log + ingestion pipeline
  incremental_merkle.ts Merkle tree engine
  csr_matrix.ts         Sparse transition matrix
  transition_policy.ts  Per-type write policy
  atom_schema.ts        Atom format + validation
  audit_log.ts          In-memory audit ring buffer
  ttl_registry.ts       Per-atom TTL tracking

src/__tests__/          Vitest test suite (38 files, 671 tests)

tools/
  harness/              Benchmark harness + SLO gate
  mcp/                  MCP server + tool catalog

integrations/
  claude-skill/         Claude Cowork skill
  claude-desktop/       Claude Desktop MCP config
  vscode/               VSCode / Claude Code integration

scripts/                Smoke tests and demo scripts
guides/                 Architecture docs, benchmark protocol
grafana/                Grafana dashboard panels

Code Style

TypeScript Strict

Strict mode enabled. No any without a comment explaining why.

Explicit Types

Prefer explicit types on function signatures.

Focused Modules

One responsibility per module. Keep source files focused.

Match Existing Style

No external formatter enforced — match the style of the file you're editing.

Tests

Every change to src/ should have test coverage. Run the relevant test file during development:

npm test -- server.test        # server API tests
npm test -- ingestion.test     # ingestion pipeline tests
npm test                       # full suite (671 tests)

All tests must pass before submitting a PR. Run npm run typecheck as well — TypeScript errors will fail CI.

Pull Request Checklist

Atom Schema Changes

Adding a new atom type requires synchronized changes across four files:

1
src/atom_schema.ts

Add to ATOM_TYPES array and update V1_PATTERN regex

2
src/transition_policy.ts

Add to TYPE_TO_INDEX (must be exhaustive)

3
src/server.ts

Update all type enum strings

4
tools/mcp/mmpm_mcp_server.ts

Update all type enum arrays
